# 背景

## 当前定位

Sopify 当前定位是：

- 先服务自用的标准 SOP skills
- 聚焦开发领域提效
- 默认开箱即用
- 同时保留让其他开发者快速接入和扩展的能力

仓库里已经有比较完整的契约层资产：

- `AGENTS.md` 里的路由与流程规则
- 分阶段的 `SKILL.md`
- 方案包与知识库模板
- 一个已经代码化的运行时样板 `scripts/model_compare_runtime.py`

当前的核心问题不是规则不够，而是这些规则还没有被一层稳定的 runtime 收口。

## 核心问题

目前很多关键行为仍然过度依赖模型在当下上下文里的临场发挥：

- 路由判断可能漂移
- 方案包结构还可能依赖模型输出形状
- replay 与知识库行为虽然有约定，但没有稳定落盘
- 状态没有被当成一等文件系统对象来管理

结果就是：

- Sopify 作为契约层已经比较强
- 但作为可重复执行的流程系统还不够稳定

## 当前发布策略

当前需要先把“工程骨架完成”和“外部可发布”这两件事拆开。

- P0: 表示 runtime skeleton 在本地已经自洽
- R0: 表示对外只发布最小切片，而不是把所有命令都包装成已完成能力

当前建议的最小发布切片是：

- `runtime-backed ~go plan`

本次明确不纳入发布承诺：

- `~go` 全流程自动执行
- `~go exec` 的真实开发桥接
- `workflow-learning` 的 runtime 化回放
- history / task state / KB bootstrap
- 任何完整 develop 自动执行能力

## 产品原则

### 1. 默认无配置也能跑主流程

用户不创建 `sopify.config.yaml`，也应该可以直接使用主路径，包括：

- 普通问答
- `~go`
- `~go plan`
- `~go exec`
- 回放 / 复盘类意图

### 2. 命令和状态由 Sopify 硬控

Sopify 必须直接掌控：

- 命令路由
- 当前所处阶段
- 运行状态落盘
- 方案产物结构
- replay 产物落盘

### 3. skill 选择仍保留模型判断空间

模型仍然负责：

- 在候选 skill 中判断哪个更相关
- 读 skill 的哪一部分
- 在既定流程内生成内容

Sopify 不应该过度替代模型的语义判断。

### 4. 扩展方式必须简单

扩展优先选择：

- 目录约定
- 轻量元信息
- 可选脚本入口

不希望把复杂配置暴露给外部用户。

### 5. runtime 必须保持薄

这个阶段做的是“流程壳”，不是“通用 agent 平台”。

因此需要：

- 单仓库
- 文件系统状态机
- 边界清晰的小模块
- 明确推迟深层架构

### 6. 上下文恢复必须由 runtime 主动完成

Sopify 不应假设模型会天然记住上一次任务上下文。

因此需要：

- 由 runtime 主动回收本地状态
- 优先恢复当前活跃流程的最小工作集
- 把跨会话续跑能力建立在文件系统状态上

### 7. 明确禁止全量自动加载 KB

`.sopify-skills/` 的价值在于可恢复，而不是每次都全量灌入模型上下文。

因此需要：

- 只按路由选择性读取
- 优先读取索引、摘要和当前活跃状态
- 禁止无条件扫描整个 `wiki/`、`history/`、`replay/`

### 8. 只吸收成熟工程模式，不复制外部形态

Sopify 可以吸收成熟产品已经验证过的工程模式，但实现必须保持自有边界。

因此需要：

- 抽象出通用模式，而不是复制外部结构和话术
- 把关键控制点沉淀到自有 runtime 模块
- 保持文档、代码、目录结构都围绕 Sopify 自身定位收口

## 当前缺口

### 缺口 A: 路由规则还没有被代码稳定接管

路由已经写在契约里，但还没有统一 runtime 模块来保证一致行为。

### 缺口 B: 状态还没有单一事实源

如果没有状态文件，Sopify 仍会过度依赖模型“记住当前处于哪一步”。

### 缺口 C: 跨会话上下文恢复还没有形成固定能力

即使项目本地已经存在 plan、history、replay、wiki，当前也还没有稳定 runtime 来决定：

- 应该先读哪些文件
- 哪些路由必须带上最小上下文
- 哪些历史信息应该被忽略

### 缺口 D: 高价值 skills 仍然缺少脚本支撑

`workflow-learning`、`templates`、`kb`、`develop` 目前主要还是“描述应该怎么做”，而不是稳定“实际做到”。

### 缺口 E: 扩展路径已有方向，但还不够操作化

现在已经明确 Sopify 未来应该能够承载和编排其他 skills，但发现方式、运行约定、产物约定还没固化。

### 缺口 F: 核心流程与外围产品层还没有明确分层

当前还没有把“主流程控制壳”和“安装、升级、通知、宿主配置修补”等外围能力明确拆开。

如果边界不先定清楚，就容易在补 runtime 的过程中把核心做重。

### 缺口 G: 安装分发与运行时代码还没有形成闭环

当前 README 的安装路径只覆盖 skills 目录，但新增 runtime 代码仍在仓库根目录。

如果不先解决分发闭环，外部用户按文档安装后拿不到完整运行时。

### 缺口 H: 对外文档承诺大于当前交付范围

当前 README 已把 `~go`、`~go plan`、`~go exec`、回放都写成用户可用能力。

但当前 runtime 更接近本地骨架而不是完整产品路径，这会直接放大发布期望和实际能力之间的落差。

## 成功标准

## P0 成功标准

P0 代表“本地工程里程碑达成”，不代表已经形成外部发布能力。

当下面这些条件成立时，P0 算完成：

- 主流程不依赖配置文件
- 命令和状态流转具备确定性
- 最小上下文回收可用于跨会话续跑
- 方案包能按固定结构生成
- replay 文件能稳定落盘
- skills 能按约定被发现
- 不会退化成全量自动加载 KB
- 入口、路由、状态、编排边界清晰且职责不重叠
- runtime 仍然足够薄、足够容易理解

## R0 最小可发布成功标准

当下面这些条件成立时，R0 算完成：

- 用户按安装文档可以拿到 runtime 所需代码
- `~go plan` 有明确 runtime 接线，而不是只依赖 prompt 契约
- runtime 结果可以渲染成稳定的用户可见输出
- 零配置下可以稳定生成 plan / state / replay 产物
- 文档承诺与真实能力一致，不再超前
- 至少有一套进仓测试和 CI 覆盖这个发布切片

## P1 成功标准

当 R0 稳定之后，下面这些条件成立时，P1 算完成：

- 最小知识库初始化和偏好持久化可用
- 历史上下文只能按需、按边界回收
- 已完成方案能归档并被索引
- 任务状态可以显式更新
- 关键 skills 开始由脚本支撑，而不只是 prompt 契约
- 核心流程仍未被外围产品层能力反向拖重

## 本蓝图明确不覆盖的内容

- 不把 Sopify 做成通用 agent operating system
- 不设计复杂插件市场
- 不在当前阶段解决所有未来 provider / platform 问题
- 不做全量自动加载知识库
- 不在第一阶段实现完整自动开发执行引擎
- 不在当前阶段优先实现安装器、升级器、通知、hooks、多 CLI 兼容层
